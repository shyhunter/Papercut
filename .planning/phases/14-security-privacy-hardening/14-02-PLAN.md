<task type="auto">
  <name>Task 1: Configure strict CSP and review capability scopes</name>
  <files>src-tauri/tauri.conf.json, src-tauri/capabilities/default.json</files>
  <action>
1. In `src-tauri/tauri.conf.json`, replace `"csp": null` with:
```json
"csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' blob: data: asset:; font-src 'self' data:; connect-src ipc: http://ipc.localhost; worker-src 'self' blob:; object-src 'none'; base-uri 'self'"
```

Key decisions:
- `script-src 'self'` — NO `'unsafe-inline'`, NO `'unsafe-eval'`. Tauri auto-relaxes in dev mode for Vite HMR.
- `style-src 'self' 'unsafe-inline'` — required for Tailwind v4 runtime style injection and React inline styles.
- `img-src 'self' blob: data: asset:` — blob: for PDF thumbnail canvas exports, data: for base64 images, asset: for Tauri asset protocol.
- `connect-src ipc: http://ipc.localhost` — allows Tauri IPC only, blocks all external HTTP/WS.
- `worker-src 'self' blob:` — PDF.js uses blob: workers.
- `object-src 'none'` — blocks plugins/embeds.
- `base-uri 'self'` — prevents base tag injection.

2. In `src-tauri/capabilities/default.json`, review `shell:allow-open`:
   - This permission allows the frontend to open URLs/paths via the OS. It's used for `reveal_in_finder` (opening file paths in Finder/Explorer).
   - The current `shell:allow-open` is unscoped — it could open arbitrary URLs in the browser.
   - Since we have zero-network-guarantee, this is acceptable for file paths but should be documented.
   - Add a comment in the JSON (via a `_comment` field or keep as-is since JSON doesn't support comments).
   - Leave `shell:allow-open` as-is — it's needed for the "reveal in Finder" feature. The CSP's `connect-src` restriction prevents the webview itself from making network calls.

3. Verify no `http:` or `https:` permission identifiers exist in default.json (should already be clean per existing privacy test).
  </action>
