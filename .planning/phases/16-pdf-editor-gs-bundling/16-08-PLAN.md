**Bug 7: Header/sidebar disappears on scroll**
The EditorTopToolbar sits in a flex column but is NOT sticky/fixed. When EditorCanvas scrolls, the toolbar scrolls away.

Fix: The EditorView layout already uses `flex flex-col h-full` with EditorTopToolbar as `flex-none`. The issue is that EditorCanvas (the middle area) has `overflow-auto` which means its scroll is contained. If the toolbar disappears, it means the entire EditorView is scrolling, not just the canvas.

Investigate the actual scroll container. The fix should ensure:
- EditorTopToolbar is always visible at the top (it already has `flex-none`, but verify the parent `h-full` chain is unbroken from root to EditorView)
- PagePanel (left) stays visible and scrolls independently
- ToolSidebar (right) stays visible and scrolls independently
- Only EditorCanvas scrolls the PDF pages

Check that `h-full` / `min-h-0` is set on all flex containers from App.tsx down to EditorView. A missing `min-h-0` or `overflow-hidden` on a flex parent causes the content to push the container height past viewport, making everything scroll.

**Bug 19: Side menu doesn't respond to window resize**
PagePanel uses fixed `PANEL_WIDTH = 180` and `COLLAPSED_WIDTH = 28`. ToolSidebar likely has similar fixed widths. These should work with flex layout.

Check if the panels use `flex-none` with a fixed `w-[180px]` — this is correct and should not cause resize issues. The real problem may be that the canvas doesn't recalculate fit-width zoom on window resize. EditorCanvas has a ResizeObserver for this — verify it fires on window resize and updates the zoom when mode is "fit-width".

Fix: Ensure ResizeObserver is attached to the canvas container (not the window) so it fires on any resize.
  </action>
  <verify>
- `npx tsc --noEmit` passes
- EditorTopToolbar stays visible when scrolling through a long PDF
- PagePanel and ToolSidebar remain visible and independently scrollable
- Resizing app window causes canvas to re-fit (if zoom mode is fit-width)
  </verify>
  <done>Toolbar and sidebars stay pinned; canvas is the only scrollable area. Window resize triggers re-layout.</done>

**Bug 8: Font/size/bold changes have no visual effect on selected text**
FormattingToolbar calls `updateTextBlock()` which updates context state. But TextEditingLayer renders text blocks with their styling. Check if TextEditingLayer reads `block.fontFamily`, `block.fontSize`, `block.color`, `block.bold`, etc. to apply CSS styles to the text overlay divs.

Likely issue: TextEditingLayer renders text blocks but does NOT apply the formatting properties as CSS styles. The contentEditable div or the block container div likely has hardcoded styles or no dynamic styling.

Fix: In TextEditingLayer, where each text block is rendered, apply dynamic CSS:
```
style={{
  fontFamily: block.fontFamily,
  fontSize: `${block.fontSize}px`,
  color: block.color,
  fontWeight: block.bold ? 'bold' : 'normal',
  fontStyle: block.italic ? 'italic' : 'normal',
  textDecoration: block.underline ? 'underline' : 'none',
}}
```

**Bug 9: Added text disappears after leaving edit mode**
When user adds text with +T and types, the text appears in the contentEditable div. On blur, it should save via `updateTextBlock()`. Check if the blur handler correctly reads the text and updates the block.

Possible issues:
- The new text block might start with empty `text` and the blur handler might not be updating it
- The contentEditable div might not be syncing `innerText` back to the block's `text` property
- The block might be re-rendered from state which still has empty text

Fix: Ensure the blur handler in TextEditingLayer reads `innerText` from the contentEditable element and calls `updateTextBlock(pageIndex, { ...block, text: innerText, isModified: true })`.

**Bug 10: Old font remnants visible after text resize**
When text is made smaller, the old larger text remains visible behind it. This is because the PDF page canvas renders the original text, and the text overlay sits on top. When the overlay text gets smaller, the original shows through.

Fix: When a text block is modified (`isModified: true`), render a white background rectangle behind the overlay text block to cover the original PDF text. This is the same approach used for deleted blocks (white-rect coverage). Apply `backgroundColor: 'white'` to modified blocks' container divs.
  </action>

- Adding new text with +T, typing, and clicking away persists the text visually
- Changing text to a smaller font does not leave remnants of the old larger text
- `npx tsc --noEmit` passes
  </verify>
  <done>Text formatting changes are visible immediately. New text persists. No remnants from old styling.</done>

**Bug 4: Drag-to-reorder not working**
PagePanel uses HTML5 drag-and-drop. Common issues:
1. `dragover` handler must call `e.preventDefault()` to allow drop — if missing, drop event never fires
2. `dragSourceRef` might not be set correctly on dragstart
3. `reorderPages()` in EditorContext might have incorrect index mapping

Debug approach:
- Check that `handleDragOver` calls `e.preventDefault()` and `e.stopPropagation()`
- Check that `handleDrop` reads the correct source and target indices
- Check that `reorderPages(fromIdx, toIdx)` in EditorContext correctly rebuilds the PDF
- Check that after reorder, the state dispatch includes updated page thumbnails
- The visual insertion indicator should show during dragover

Also verify: after reorder, does the main canvas (EditorCanvas) re-render with the new page order? The canvas reads `pdfBytes` from context — if `reorderPages()` updates `pdfBytes`, the canvas should re-render.

**Bug 5: Add pages from other PDFs crashes to black screen**
PagePanel's "From PDF file..." opens a file dialog, reads bytes, calls `addPagesFromPdf(insertAfter, bytes)`.

Likely crash causes:
1. The file dialog may return an incompatible path or the readFile may fail silently
2. `PDFDocument.load()` on the source PDF may throw if it's encrypted or malformed — needs try/catch
3. After adding pages, the INIT dispatch may have mismatched page count vs pages array length
4. The new pages have empty `PageEditState` — if EditorCanvas expects page dimensions from state but they're missing, it could crash

Fix:
- Wrap `addPagesFromPdf()` in try/catch and show a toast/alert on failure instead of crashing
- After copying pages, verify the pages array length matches the new PDF page count
- Ensure new page states include correct dimensions from the source PDF pages
- Add error boundary protection around the operation

- Click "From PDF file..." → select a valid PDF → pages are inserted without crash
- Click "From PDF file..." → select an invalid file → error shown, no black screen
- `npx tsc --noEmit` passes
  </verify>

**Bug 1: Crash on 256-page PDF**
EditorCanvas renders all pages but uses virtualized rendering (only current +/-2 pages rendered to canvas). The crash likely comes from:
1. Initial PDF load: `PDFDocument.load(pdfBytes)` + extracting 256 page dimensions may be slow but shouldn't crash
2. Creating 256 PageEditState objects at once in the INIT dispatch
3. The IntersectionObserver watching 256 page elements simultaneously
4. Memory: rendering even placeholder divs for 256 pages shouldn't crash, but check if any code creates 256 canvas elements simultaneously

Fix approach:
- Add a loading state during PDF initialization for large files (show spinner instead of rendering immediately)
- Ensure the virtualization window is strict: only render canvas for pages within +/-2 of currentPage, use lightweight placeholders (just height/width div) for all others
- If the page extraction loop (getting dimensions from pdfjs) is synchronous and blocking, batch it with requestAnimationFrame breaks
- Consider: lazy page dimension extraction — only extract dimensions for visible pages, estimate others from first page

**Bug 6: Close button (X) not working in editor**
EditorView registers `onCloseRequested` from `@tauri-apps/api/window`. Check:
1. Is the event listener actually being registered? (useEffect dependency array)
2. Does the confirm dialog block the close? If user clicks "Don't save", the close should proceed
3. The `beforeunload` listener may conflict with Tauri's close handler
4. `event.preventDefault()` on the close request might be called unconditionally instead of only when dirty

Fix: Ensure `onCloseRequested` handler:
- Only prevents close when isDirty is true
- Shows confirm dialog with three options: Save, Don't Save, Cancel
- "Don't Save" allows close to proceed (does NOT call event.preventDefault())
- "Cancel" prevents close
- "Save" saves and then allows close

Also check: The Tauri window might be configured with `"closable": false` in tauri.conf.json — verify it's true.
  </action>
  <verify>
- Open a 200+ page PDF → editor loads without crash (may show loading state)
- Scroll through all pages → no crash or excessive memory
- Click window X button with no changes → window closes
- Click window X button with unsaved changes → confirmation dialog appears → all three options work
- `npx tsc --noEmit` passes
  </verify>
  <done>Large PDFs load without crash. Window X button works correctly with unsaved-changes guard.</done>
</task>
