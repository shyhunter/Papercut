<task type="auto">
  <name>Task 2: Update find_ghostscript() to prefer sidecar binary</name>
  <files>src-tauri/src/lib.rs</files>
  <action>
Refactor `find_ghostscript()` in lib.rs:

1. First, try to resolve the sidecar binary using Tauri's `app.path().resolve("binaries/gs", BaseDirectory::Resource)` or equivalent sidecar resolution. Since find_ghostscript is called from Tauri commands that have access to `app_handle`, pass the app handle (or a resolved sidecar path) to find_ghostscript.

   Actually, the simpler approach: use `std::env::current_exe()` to find the app binary location, then resolve `../Resources/binaries/gs` (macOS) or `binaries/gs` (Linux/Windows) relative to it. This is the standard Tauri sidecar resolution pattern.

   Alternatively, use the `tauri::api::process::Command::new_sidecar("binaries/gs")` pattern if available in Tauri v2. Check Tauri v2 docs for the correct sidecar invocation.

2. If sidecar binary exists and is executable, return its path.
3. If sidecar not found, fall back to the current system PATH lookup (existing behavior).
4. Update all callers of find_ghostscript() — they currently use `std::process::Command::new(&gs_bin)`. If using Tauri sidecar Command, the invocation pattern changes to `Command::new_sidecar("gs")`.

   The key insight: in compress_pdf and other GS commands, replace `std::process::Command::new(&gs_bin)` with Tauri's sidecar Command when the sidecar is available. Keep std::process::Command as fallback for system GS.

3. Ensure the existing `cancel_processing` / kill flow still works with the sidecar process.
  </action>

`cargo check --manifest-path src-tauri/Cargo.toml` compiles without errors.
Run the app with `npm run dev` (from project root) and test PDF compression — it should use the bundled GS binary.
  </verify>

</task>
