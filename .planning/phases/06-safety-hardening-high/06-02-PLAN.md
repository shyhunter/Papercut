<objective>
Add cancellation support to long-running processing operations so users can abort mid-flight and recover gracefully without restarting the app.

Purpose: A large PDF sent through Ghostscript can run for 30+ seconds. Without cancellation, the user is stuck waiting or must force-quit. This plan adds a Cancel button (Configure step, visible during processing) and a Cancelled recovery state (Compare step) for both PDF and image pipelines.
Output: Updated Rust compress_pdf with tokio::select! cancellation, new cancel_processing Tauri command, cancel() on both processor hooks, Cancel button on both Configure steps, Cancelled state on CompareStep.
</objective>

    4. Add `cancel_processing` Tauri command:
       ```rust
       #[tauri::command]
       fn cancel_processing(state: tauri::State<CancelState>) {
           let mut guard = state.sender.lock().unwrap();
           if let Some(tx) = guard.take() {
               let _ = tx.send(());
           }
       }
       ```
       Register it in invoke_handler: `greet, process_image, compress_pdf, cancel_processing`.

    5. Update `compress_pdf` signature to accept `state: tauri::State<'_, CancelState>`:
       ```rust

       async fn compress_pdf(
           app: tauri::AppHandle,
           state: tauri::State<'_, CancelState>,
           source_path: String,
           preset: String,
       ) -> Result<tauri::ipc::Response, String> {
       ```

    6. Inside `compress_pdf`, before calling GS, create a oneshot channel and store the sender:
       ```rust
       let (tx, rx) = oneshot::channel::<()>();
       {
           let mut guard = state.sender.lock().unwrap();
           *guard = Some(tx);
       }
       ```

    7. Replace the `compress_pdf_with_gs` call with a `tokio::select!`:
       ```rust
       tokio::select! {
           result = compress_pdf_with_gs(&app, &source_path, &tmp_path_str, &preset) => {
               result?;
           }
           _ = rx => {
               // Cancelled ‚Äî clean up temp file and return a distinct error string
               let _ = std::fs::remove_file(&tmp_path);
               return Err("CANCELLED".to_string());
           }
       }
       ```
       Note: The GS `.output().await` in compress_pdf_with_gs is not truly interruptible via tokio::select! alone because `.output()` waits for process exit. To achieve real process kill: refactor compress_pdf_with_gs to use `.spawn()` (returns `CommandChild`) and store the child in a `Mutex<Option<Child>>` OR use `tokio::time::timeout` as a fallback.

       Pragmatic approach for this codebase: use `tokio::select!` with `tokio::time::timeout` + a `std::sync::atomic::AtomicBool` cancel flag checked at strategic points. For the GS subprocess specifically, since `.output()` blocks until the process exits, the most reliable kill is: spawn the GS process separately (using `std::process::Command::spawn()` in a blocking thread), store the child PID in state, and call `child.kill()` from cancel_processing. However, tauri-plugin-shell's sidecar API returns a `CommandChild` with `.kill()` when using `.spawn()`.

       IMPLEMENTATION DECISION: Use tauri-plugin-shell's `.spawn()` API (not `.output()`) to get a `CommandChild`. Collect stdout/stderr via the event stream. Store the `CommandChild` in a separate Mutex in state. cancel_processing calls `.kill()` on the stored child.

       Updated state struct:
       ```rust
       use tauri_plugin_shell::process::CommandChild;

       struct ProcessState {
           cancel_sender: Mutex<Option<oneshot::Sender<()>>>,
           gs_child: Mutex<Option<CommandChild>>,
       }
       ```

       In compress_pdf: spawn with `.spawn()`, get `(mut rx_events, child)`. Store child in state.gs_child. Read events in a loop collecting stdout/stderr until CommandEvent::Terminated. Use tokio::select! between the event loop and the cancel rx.

       In cancel_processing: lock gs_child, call `.kill()` on the child if present; lock cancel_sender, send cancellation.

       The GS process writes its output to the tmp_path file ‚Äî so we do NOT need to collect stdout bytes; we read tmp_path after the process exits. The event loop just waits for termination.

    8. Add the `tokio` dependency to Cargo.toml if needed:
       Check with `grep -n "tokio" src-tauri/Cargo.toml`. If absent, add:

       ```toml
       tokio = { version = "1", features = ["sync"] }
       ```
       tauri 2 uses tokio internally ‚Äî if there's a version conflict, use `tauri::async_runtime::spawn` patterns instead and use `std::sync::mpsc` for the cancel channel.
