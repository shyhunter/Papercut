  <action>
    1. In `lib.rs`, find all `.sidecar("gs")` call sites (at least 5: compress_pdf, convert_to_pdfa, repair_pdf, protect_pdf, unlock_pdf).
    2. Replace each `.sidecar("gs")` call with a system binary resolution approach:
       - Create a `find_ghostscript()` helper function that:
         - On macOS/Linux: checks `which gs` via `std::process::Command`
         - On Windows: checks `where gswin64c` then `where gs`
         - Also checks the detect_converters data if available
         - Returns `Result<PathBuf, String>` with a clear error message if GS not found
       - Replace each `app.shell().sidecar("gs")` with `std::process::Command::new(gs_path)` (or Tauri shell command with the resolved path)
    3. Ensure the existing cancellation pattern (spawn + kill via ProcessState mutex) still works with the new approach.
    4. Remove `externalBin` entries for GS from `tauri.conf.json`.
    5. Remove any GS-related sidecar entries from `Cargo.toml` if present.
    6. Keep the `shell:allow-execute` capability since system commands still need it.

    IMPORTANT: The detect_converters system from Phase 14 already knows how to find system GS. Reuse that detection logic if possible rather than duplicating it.
  </action>
  <verify>
    `cd src-tauri && cargo check` passes without errors.
    `grep -r "sidecar" src-tauri/src/lib.rs` returns no GS-related sidecar references.
    `grep -r "externalBin" src-tauri/tauri.conf.json` returns no GS entries.
  </verify>
  <done>All GS invocations use system-installed binary via PATH lookup. No sidecar references remain. Cargo check passes.</done>
